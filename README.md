# Задание 7. Тестирование работоспособности системы

## Описание

В рамках задания выполнено тестирование работоспособности ML-сервиса как единой системы.

Проверялись не отдельные функции, а сквозные пользовательские сценарии, включающие:

- регистрацию и авторизацию пользователей;
- работу с балансом;
- выполнение ML-запросов;
- списание кредитов;
- обработку ошибок;
- сохранение истории транзакций;
- сохранение истории ML-запросов.

Цель задания — подтвердить, что все компоненты проекта корректно работают вместе:

- FastAPI REST API;
- бизнес-логика;
- база данных PostgreSQL;
- RabbitMQ;
- worker-процессы;
- инфраструктура Docker Compose.

---

## Архитектура системы

Проект состоит из следующих частей:

- `FastAPI app` — REST API и web-интерфейс;
- `PostgreSQL` — хранение пользователей, баланса, транзакций и истории предсказаний;
- `RabbitMQ` — очередь задач для ML-обработки;
- `worker-1`, `worker-2` — асинхронная обработка ML-задач;
- `nginx` — reverse proxy.

---

## Что проверялось

### 1. Работа с пользователями

Проверены сценарии:

- создание нового пользователя;
- авторизация по логину;
- повторная авторизация;
- ошибка авторизации при неверном пароле.

### 2. Работа с балансом

Проверены сценарии:

- получение начального баланса;
- пополнение баланса;
- проверка обновлённого баланса после пополнения.

### 3. Списание кредитов

Проверены сценарии:

- успешное списание после успешного ML-запроса;
- запрет выполнения ML-запроса при недостаточном балансе;
- отсутствие списания при ошибке ML-обработки.

### 4. ML-запросы

Проверены сценарии:

- успешная отправка данных на предсказание;
- получение результата предсказания;
- обработка некорректных входных данных;
- обработка частично валидных данных.

### 5. История операций

Проверены сценарии:

- сохранение истории транзакций;
- сохранение истории ML-запросов;
- соответствие истории ранее выполненным действиям.

## Автотесты

Для проверки системы реализованы интеграционные тесты в файле:

~~~
app/tests/test_task_7.py
~~~

**Тесты используют:**

* `FastAPI TestClient` для проверки REST API;
* отдельную тестовую SQLite-базу;
* подмену публикации сообщений в RabbitMQ;
* прямой вызов `process_prediction_task(...)` для проверки полного цикла обработки задачи.

Такой подход позволяет проверить бизнес-сценарии из задания без зависимости от внешнего окружения во время запуска тестов.

## Что удалось выявить во время тестирования

Во время разработки тестов были найдены и исправлены следующие проблемы:

### 1. Ошибка сериализации `RequestValidationError`

При отправке некорректного запроса API должен был возвращать корректный ответ `422`, но вместо этого происходила внутренняя ошибка сериализации.

Причина была в том, что в `exc.errors()` внутри `ctx` мог содержаться объект `ValueError`, который не сериализуется в JSON напрямую.

После исправления обработчик ошибок начал корректно возвращать пользователю JSON-ответ с кодом `422`.

### 2. Ошибка в тестовом helper для обработки задач

Во время написания тестов был допущен дефект в вспомогательной функции `process_task(...)`, из-за которого часть сценариев падала не по причине проблем бизнес-логики, а из-за ошибочного кода самого теста.

После исправления helper-функции тесты начали корректно проверять именно логику приложения.

## Запуск проекта

Остановить и удалить старые контейнеры и volume:

```bash
docker compose down -v
```

Запустить систему:

```bash
docker compose up --build
```

---

## Проверка, что сервисы запущены

Проверить контейнеры:

```bash
docker compose ps
```

Проверить health endpoint:

```bash
curl http://localhost/health
```

## Запуск автотестов

Запуск тестов внутри контейнера приложения:

```bash
docker compose exec app python -m pytest -v tests
```

Краткий вариант:

```bash
docker compose exec app python -m pytest -q tests
```
